How we designed, built, and deployed a cloud-based fitness coaching platform as a 4-week capstone project using Azure, Python, Terraform, and GitHub Actions.

1. Project Purpose

MacroFit Coach is a cloud-based fitness coaching web application built as the final capstone project for our Cloud Computing course. The goal was to design and deploy a real-world web application that demonstrates practical cloud skills including static hosting, serverless APIs, NoSQL databases, Infrastructure as Code, CI/CD automation, DNS, HTTPS, monitoring, and security.

Rather than building a generic demo, we chose a fitness coaching platform because it gave us meaningful sections to build — a macro nutrition tracker, a workout log, and a coach communication portal — all backed by cloud infrastructure. This made the project feel like a real product and gave us more to talk about during the final demo.

Project Name: MacroFit Coach — Track Your Macros. Log Your Workouts. Stay Connected With Your Coach.

3. How the Visitor Counter Works

The visitor counter is the most technically interesting component of the project. It demonstrates the full serverless cloud pattern: a browser calls an API, the API reads and writes to a database, and the result is returned and displayed.

Architecture Flow

Step 1 — Browser loads the page.
The static HTML/CSS/JS files are served from Azure Blob Storage. The site is delivered over HTTPS through Azure CDN.

Step 2 — JavaScript calls the API.
counter.js sends a POST request to the Azure Functions endpoint. POST was chosen to signal "this is a new visit" vs GET (read-only).

Step 3 — Azure Function runs Python code.
The function uses the azure-cosmos SDK to connect to Cosmos DB. It reads the current count document, increments the value by 1, and saves it using upsert_item().

Step 4 — JSON response returned.
The function returns { "count": 1248, "status": "ok" }. CORS headers are set to allow requests from the frontend domain only.

Step 5 — Counter animates on screen.
JavaScript uses requestAnimationFrame with an ease-out cubic function to smoothly count from 0 to the returned value. If the API is unreachable, a cached value from localStorage is used as a fallback.

Cosmos DB Structure

We use a single Cosmos DB SQL API container named visitor-counter with the partition key /partitionKey. The counter is stored as one document: 

{
  "id": "global-visitor-count",
  "partitionKey": "counter",
  "count": 1248
}

This structure is intentionally simple. In a future version, the database would also hold tables for macro entries, workout logs, and client messages — each linked to a user ID.

4. Infrastructure as Code (Terraform)

All Azure resources are defined using Terraform so they can be created and destroyed repeatedly, consistently, and without clicking through the portal. This makes the project reproducible and demonstrates cloud automation skills.

Our Terraform configuration (iac/terraform/main.tf) provisions:

  • An Azure Resource Group to contain all resources
  • An Azure Storage Account with static website hosting enabled
  • An Azure Cosmos DB account with free tier + SQL API
  • A Cosmos DB database and container for the visitor counter
  • An Azure App Service Plan (consumption / serverless)
  • An Azure Linux Function App running Python 3.11
  • Application Insights connected to a Log Analytics workspace
  • A monthly budget alert at 80% and 100% of the spending cap

Key benefit of IaC: Any team member can run terraform apply and have the entire infrastructure running in under 5 minutes — no manual setup required.

5. CI/CD Automation (GitHub Actions)

Every push to the main branch triggers our GitHub Actions pipeline, which automatically validates, deploys, and verifies the project. This eliminates manual FTP uploads and ensures every commit goes live consistently.

Our pipeline (.github/workflows/deploy.yml) has four jobs:

  • Job 1 — Lint & Validate: Checks that all HTML, CSS, and JS files exist. Runs flake8 on the Python API code to catch syntax errors.
  • Job 2 — Deploy Static Site: Uses the Azure CLI to upload all HTML, CSS, and JS files to the Azure Blob Storage $web container.
  • Job 3 — Deploy Azure Function: Installs Python dependencies and deploys the visitor counter API to Azure Functions using the official Azure/functions-action.
  • Job 4 — Deployment Summary: Prints a summary to the GitHub Actions log confirming the branch, commit SHA, and actor who triggered the deploy.

GitHub Secrets are used to store all sensitive credentials (AZURE_CREDENTIALS, STORAGE_ACCOUNT_NAME, FUNCTION_APP_KEY) — they are never exposed in the source code.

6. DNS and HTTPS Setup

The site is served over HTTPS using Azure CDN with a custom domain. Here is the step-by-step process we followed:

  • Registered a custom domain (e.g., macrofitcoach.com) through a registrar
  • Created an Azure CDN profile and endpoint pointing at the Blob Storage static site
  • Added a CNAME record in the domain registrar DNS settings pointing to the CDN endpoint
  • Added the custom domain to the CDN endpoint in the Azure Portal
  • Enabled the free Azure-managed TLS/HTTPS certificate on the CDN endpoint
  • Configured an HTTP → HTTPS redirect rule so all traffic uses TLS

Result: the site loads over https://macrofitcoach.com with a valid certificate, an A+ security rating, and automatic certificate renewal.

7. Monitoring, Logging & Alerts

Azure Application Insights is connected to the Azure Function App and tracks:

  • Every API request to the visitor counter (response time, status code, success/failure)
  • Function execution errors with full stack traces
  • Live metrics dashboard viewable in the Azure Portal
  • Dependency tracking for Cosmos DB calls

Logs are stored in a Log Analytics Workspace with a 30-day retention window. We can query logs using Kusto Query Language (KQL) to analyze traffic patterns and debug issues.

A monthly budget alert is configured using Terraform to email the team when Azure spending reaches 80% and 100% of our cap. This prevents unexpected charges and demonstrates cloud cost management awareness.

8. Security Practices

We followed these security best practices throughout the project:

  • No secrets in source code — all credentials are in GitHub Secrets or Azure App Settings (environment variables)
  • Least-privilege IAM — the CI/CD service principal has only the permissions needed to deploy
  • CORS configured — the Azure Function only allows requests from the frontend domain
  • HTTPS everywhere — all traffic is encrypted via Azure CDN + managed TLS certificate
  • XSS prevention — all user inputs rendered in JavaScript are HTML-escaped before DOM insertion
  • Input validation — form inputs use type, min, and max HTML attributes plus server-side validation
  • Function authentication — Azure Function requires a function key for API access

9. Challenges & Lessons Learned

Challenge 1: CORS Configuration

When we first deployed the Azure Function, the browser blocked all API calls with a CORS error. We had to configure cors.allowed_origins in both the Terraform function app settings and in our Python function code. We also needed to handle the preflight OPTIONS request correctly. Lesson: Always test your API from the actual frontend domain, not just Postman.

Challenge 2: Terraform State Management

On the first terraform apply, we forgot to configure a remote state backend. This meant each team member's local state file was out of sync. We resolved this by adding an Azure Blob Storage backend for Terraform state so the state file is shared and locked during applies. Lesson: Always configure remote state before working as a team.

Challenge 3: Cosmos DB Free Tier Limit

Azure only allows one Cosmos DB free tier account per subscription. Two team members tried to deploy in the same subscription and got an error. We resolved this by designating one subscription for the project and sharing access. Lesson: Understand subscription-level limits before starting.

Challenge 4: GitHub Actions Azure Login

Configuring the AZURE_CREDENTIALS secret for GitHub Actions required creating a service principal with the correct role assignments. The JSON format for the secret is specific and easy to get wrong. Lesson: Follow the official Azure/login action documentation exactly, and test the login step in isolation first.
